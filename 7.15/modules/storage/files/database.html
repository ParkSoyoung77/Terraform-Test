데이터베이스 (RDS)

서비스의 상태를 보관하는 계층입니다. 컴퓨팅 계층과 달리 함부로 여러 대를 늘려 쓸 수 없기 때문에, 가용성과 읽기 성능을 별도로 설계해야 합니다.

LAYER 03 / 03

3.1 RDS와 관리형 데이터베이스

RDS(Relational Database Service)는 MySQL, PostgreSQL 같은 관계형 데이터베이스를 AWS가 대신 운영해주는 서비스입니다. 백업, 패치, 장애 감지 같은 운영 부담을 줄여주는 대신, 인스턴스에 직접 접속해 OS 레벨을 만지는 자유도는 낮아집니다.

프라이빗 배치

데이터베이스는 외부에 노출될 이유가 없으므로 프라이빗 서브넷에 두고, 애플리케이션 계층에서만 접근하도록 보안 그룹을 제한합니다.

백업 보존 기간

자동 백업을 얼마나 보관할지 설정합니다. 0으로 두면 자동 백업이 꺼지며, 읽기 복제본을 만들려면 반드시 0보다 커야 합니다.

3.2 Multi-AZ: 장애에 대비한 이중화

Multi-AZ를 활성화하면 다른 가용 영역에 대기 인스턴스(Standby)가 동기식으로 복제됩니다. 주 인스턴스에 장애가 발생하면 자동으로 대기 인스턴스로 전환되어, 별도 조치 없이도 서비스가 이어집니다.

NORMAL 주 인스턴스가 요청을 처리하고, Standby는 동기식으로 데이터를 복제합니다.
FAILOVER 주 인스턴스에 장애가 발생하면 DNS 엔드포인트가 Standby로 자동 전환됩니다.
RECOVER 기존 주 인스턴스는 복구된 뒤 새로운 Standby 역할을 맡습니다.

3.3 읽기 복제본과 연결 관리

읽기 요청이 많은 서비스는 Read Replica로 조회 트래픽을 분산합니다. Multi-AZ가 "장애 대비"라면, Read Replica는 "성능 확장"에 가깝습니다. 또한 RDS Proxy를 두면 애플리케이션(특히 Lambda처럼 연결이 짧고 잦은 환경)이 매번 새 연결을 맺는 대신 커넥션 풀을 재사용할 수 있습니다.

구성 요소목적동기화 방식
Multi-AZ Standby장애 발생 시 자동 전환동기식
Read Replica읽기 트래픽 분산비동기식
RDS Proxy연결 재사용, 커넥션 폭주 방지해당 없음 (연결 계층)

3.4 실무에서 자주 확인하는 지점

CHECKPOINT Lambda에서 RDS로 접속이 안 될 때는 ① 둘이 같은 VPC/서브넷 그룹에 있는지, ② 보안 그룹이 Lambda 쪽에서 들어오는 3306 포트를 허용하는지, ③ 비밀번호를 Secrets Manager로 조회하는 구조라면 해당 API 호출이 나갈 아웃바운드 경로(NAT 또는 VPC 엔드포인트)가 있는지 순서로 확인하는 것이 일반적입니다.
