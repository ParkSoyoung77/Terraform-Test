01 · Network Layer

네트워크 (VPC)

모든 리소스가 통신할 경로를 미리 설계하는 계층입니다. 어떤 서브넷이 외부와 직접 연결되는지, 내부 리소스가 어떻게 외부로 나가는지를 결정합니다.

LAYER 01 / 03

1.1 VPC란

VPC(Virtual Private Cloud)는 AWS 계정 안에 논리적으로 격리된 가상 네트워크입니다. IP 대역(CIDR)을 직접 정하고, 그 안을 서브넷으로 나누어 리소스를 배치합니다. 하나의 VPC는 여러 가용 영역(AZ)에 걸쳐 구성할 수 있어, 특정 AZ에 장애가 생겨도 다른 AZ의 서브넷은 영향을 받지 않도록 설계할 수 있습니다.

퍼블릭 서브넷

인터넷 게이트웨이로 향하는 라우팅이 걸려 있어, 외부와 직접 통신이 필요한 리소스(로드밸런서, Bastion 등)를 배치합니다.

프라이빗 서브넷

외부에서 직접 접근할 수 없습니다. 애플리케이션 서버나 데이터베이스처럼 외부에 노출되면 안 되는 리소스를 배치합니다.

1.2 라우팅 테이블과 트래픽 방향

서브넷 자체는 "퍼블릭"이라는 속성을 갖지 않습니다. 실제로는 해당 서브넷에 연결된 라우팅 테이블이 트래픽을 어디로 보내는지에 따라 퍼블릭/프라이빗이 결정됩니다.

IN 외부 요청이 인터넷 게이트웨이(IGW)를 통해 VPC로 들어옵니다.
ROUTE 라우팅 테이블이 목적지 서브넷을 결정합니다. 0.0.0.0/0 → IGW 규칙이 있으면 퍼블릭 서브넷입니다.
OUT 프라이빗 서브넷의 리소스가 외부로 나가야 할 때는 NAT를 거쳐야 합니다.

1.3 NAT: 나가는 길은 열고 들어오는 길은 막는다

NAT(Network Address Translation)는 프라이빗 서브넷의 리소스가 인터넷 쪽으로 요청을 보낼 수 있게 해주지만, 반대로 외부에서 그 리소스로 먼저 접속하는 건 막습니다. 소프트웨어 업데이트나 외부 API 호출처럼 "나가는 트래픽"만 필요한 경우에 씁니다.

방식운영 부담특징
NAT Gateway낮음 (완전 관리형)AWS가 관리, 트래픽 기준 과금, 별도 설정 거의 불필요
NAT Instance높음 (직접 관리)EC2로 직접 구성, Source/Dest Check 해제와 IP 포워딩 설정이 필요

1.4 실무에서 자주 확인하는 지점

CHECKPOINT 프라이빗 서브넷의 리소스가 외부와 통신이 안 될 때는 ① 라우팅 테이블이 NAT를 가리키는지, ② NAT 앞단 보안 그룹이 필요한 포트를 열어뒀는지, ③ NAT 인스턴스 방식이라면 Source/Dest Check가 꺼져 있는지 순서로 확인하는 것이 일반적입니다.
